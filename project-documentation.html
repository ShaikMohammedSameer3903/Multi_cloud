Executive Summary & Project Overview

The Azure Healthcare Platform is an enterprise-grade cloud operations dashboard designed to manage, monitor, and govern a HIPAA-compliant healthcare infrastructure deployed on Microsoft Azure. The platform provides real-time visibility into deployed resources, security posture, backup health, cost analytics, and compliance governance for a simulated healthcare organization.

🎯 Project Objective Design, deploy, and operate a production-ready Azure Landing Zone with a Level 3 Enterprise Operations Center dashboard that demonstrates live integration with Azure Resource Manager, Azure Monitor, Recovery Services, Key Vault, and Cost Management APIs.

Project Scope

  • Infrastructure as Code: Azure Bicep templates for automated, repeatable infrastructure provisioning
  • Live Dashboard: React + TypeScript frontend with Node.js backend fetching real-time data from Azure SDKs
  • Security: Microsoft Entra ID authentication with JWT validation, RBAC roles, and PIM governance
  • Monitoring: Azure Monitor metric alerts with tuned thresholds to eliminate false positives
  • Disaster Recovery: Geo-Redundant backups with automated restore dry-run validation scripts
  • Governance: Azure Policy enforcement, resource locks, and tagging compliance
  • Cost Controls: Budget alerting with tiered escalation thresholds

All infrastructure resources are defined declaratively using Azure Bicep templates. This ensures repeatable, auditable, and version-controlled provisioning of the entire landing zone.

Core Bicep Template — iac/main.bicep

The main Bicep template provisions the following resources:

Key Vault Configuration (Bicep Snippet)

resource keyVault 'Microsoft.KeyVault/vaults@2023-02-01' = {
  name: keyVaultName
  location: location
  properties: {
    sku: { family: 'A', name: 'premium' }
    tenantId: subscription().tenantId
    enableSoftDelete: true
    softDeleteRetentionInDays: 90
    enablePurgeProtection: true
    enableRbacAuthorization: true    // Modern Azure IAM standard
    networkAcls: {
      bypass: 'AzureServices'
      defaultAction: 'Deny'         // Zero-trust network model
    }
  }
}

Security Middleware Stack

  • Helmet: Sets security HTTP headers (CSP, X-Frame-Options, etc.)
  • CORS: Whitelist-based origin validation for localhost and Azure Static Web Apps
  • Rate Limiter: 200 requests per 15 minutes per IP
  • JWT Validation: Verifies Microsoft Entra ID RS256 tokens via JWKS endpoint
  • Audit Logger: Logs IP, user identity, method, URL, status code, and response duration

Monitoring, Alerting & Alert Tuning RCA

Incident Background

🚨 INC-20260601-BF — Alert Noise Fatigue Masking SQL Database Backup Failure During UAT, a persistent backup failure on PatientRecordsDB was masked by 186 daily transient alerts, causing a 36-hour backup gap and HIPAA compliance risk.

Root Cause Analysis (5 Whys)

  1. Why was the backup failure ignored? — Operations team silenced notifications
  2. Why were notifications silenced? — Severe alert fatigue from 180+ daily false positives
  3. Why so many false alarms? — Alert threshold was set to 1 failure, evaluated every 1 minute
  4. Why did single failures occur? — Transient network blips caused single-attempt drops
  5. Why were rules designed this way? — Default UAT template without production tuning

Alert Tuning Results

Metric Before Tuning (UAT) After Tuning (Production) Improvement
False-Positive Alarms186 alerts / day12 alerts / day93.5% Reduction
Severity Distribution100% Critical20% Critical / 80% WarningClean Escalation
True Positive Capture100%100%Zero Missed Failures
Incident Response SLA> 8 Hours< 15 Minutes96.8% Speedup

Tuning Parameters Applied

Parameter Before (Default) After (Tuned)
Evaluation Frequency1 Minute (PT1M)15 Minutes (PT15M)

Backup & Disaster Recovery

Backup Policy Configuration

Protected Scope:PatientRecordsDB (Primary SQL Database) + Production VMs Frequency:Full weekly (Sunday 00:00 UTC) + Daily incremental (02:00 UTC) Retention:52 weekly recovery points + 60 monthly backups Redundancy:Geo-Redundant Storage (GRS) with Cross-Region Restore Vault Security:Soft Delete enabled (14-day hold), Purge Protection ON

Automated Restore Dry-Run Pipeline

The restore validation pipeline runs daily to ensure backup integrity:

  1. Provision Container: Build an isolated sandbox subnet (sub-hc-db-restore-test)
  2. Retrieve Log Slices: Pull transaction log slices from Recovery Services Vault
  3. Execute Restore: Restore database instance to the ephemeral container
  4. Integrity Check: Run DBCC CHECKDB consistency verification
  5. MD5 Hash Validation: Compare source backup hash with restored database hash

Runbook: OP-01 — Recovery Services Backup Failure

Severity: P1 — HIPAA compliance breach if unrecovered within 24 hours SLA Resolution Time: 2 Hours | Assigned: On-Call Database Administrator
  1. Activate Backup Contributor role via PIM with justification
  2. Retrieve failed job details: Get-AzRecoveryServicesBackupJob -Status Failed
  3. Execute sandbox restore verification: .\scripts\validate-backup.ps1
  4. Identify and clear resource blockers (log partition space, file locks)
  5. Trigger immediate manual backup: Backup-AzRecoveryServicesBackupItem

Runbook: OP-02 — PIM Access Policy Violations

Severity: P1 — Security Threat SLA Resolution Time: 30 Minutes | Assigned: SOC Incident Responder
  1. Run KQL queries on Key Vault audit logs to isolate compromised caller
  2. Revoke active tokens: Revoke-AzureADUserAllRefreshToken
  3. Rotate all Key Vault secrets and certificates immediately
